# Section 1: Introduction

## 1.1 The Evaluation Bottleneck in Medical AI

The past decade has produced remarkable advances in machine learning applied to
clinical tasks. Models trained on retinal images diagnose diabetic retinopathy
at specialist-level sensitivity [cite]; transformer architectures extract
structured diagnoses from unstructured notes with accuracy exceeding human
coders in constrained settings [cite]; large language models pass medical
licensing examinations by substantial margins [cite]. And yet the translation
of these capabilities into systems that reliably improve patient outcomes
remains sparse, contested, and slower than the rate of model development
would suggest [cite].

We argue that this gap is not primarily a deployment problem. It is an
evaluation problem. The metrics used to decide whether a model is ready for
clinical use are failing — quietly, systematically, and in ways that are
difficult to detect without a framework specifically designed to surface them.

This paper introduces **stratum-eval**, an open-source evaluation framework
whose central design principle is that a metric which cannot state what it
cannot measure should not be used to make clinical deployment decisions. We
accompany this framework with a taxonomy of evaluation failures observed
in published medical AI benchmarks, a set of enforceable metric standards,
and a suite of diagnostic tests that can be applied to any evaluation pipeline.

## 1.2 A Taxonomy of Evaluation Failures

We identify eight canonical evaluation failure modes, organized along two axes.

The **source axis** classifies where the failure originates:
- **Data** failures arise from properties of the evaluation set that are
  miscommunicated or unexamined.
- **Metric** failures arise from a mismatch between what a measure computes
  and what it is claimed to measure.
- **Deployment** failures arise from a gap between evaluation conditions and
  the conditions under which the model will operate.
- **Normative** failures arise from underspecified or contested value
  commitments embedded in the evaluation design.

The **detectability axis** classifies whether the failure is observable
without additional instrumentation:
- **Manifest** failures produce anomalous numbers that an attentive reviewer
  can detect (e.g., AUROC = 1.0 on a held-out set from the same institution
  as training data).
- **Latent** failures produce plausible numbers that conceal the problem
  (e.g., macro-AUROC of 0.85 that hides a subgroup where AUROC = 0.51).

**Table 1. Failure Taxonomy (F1–F8)**

| ID | Name | Source | Detectability | Definition | Canonical Example | stratum-eval Diagnostic |
|----|------|--------|--------------|------------|-------------------|------------------------|
| F1 | Label Endogeneity | Data | Latent | Labels are generated by a process that is causally downstream of the model's training signal, so the evaluation measures the model's influence on labeling rather than its accuracy against ground truth | Radiology AI trained on radiologist reports evaluated against reports produced by the same radiologists who saw the AI output | Annotator-blinding audit; label provenance check |
| F2 | Uncertainty Conflation | Metric | Latent | A metric aggregates aleatoric and epistemic uncertainty into a single score, obscuring whether the model's errors arise from irreducible noise or from knowledge gaps that additional training could close | Brier score reported without decomposition; confidence interval on a model score interpreted as epistemic when it reflects label noise | Uncertainty decomposition test; calibration-by-subgroup |
| F3 | Label Distribution Collapse | Data | Manifest (often overlooked) | The evaluation set's class distribution has been manipulated (deliberately or by inclusion criteria) so that it no longer reflects the distribution under which the model will be used, making prevalence-sensitive metrics (PPV, NPV, F1) uninterpretable | Balanced test set used to report F1 for a condition with 2% clinical prevalence; metrics not re-calibrated to deployment prevalence | Prevalence mismatch detector; operating-point shift analysis |
| F4 | Spurious Invariance | Metric | Latent | A metric is mathematically invariant to a model property that is clinically important, making it impossible to distinguish models that differ on that property using the metric alone | AUROC is invariant to monotone score transformations, so two models with identical AUROC can have arbitrarily different calibration and thus different decision thresholds | Calibration audit; threshold-sensitivity analysis |
| F5 | Reasoning Opacity | Deployment | Latent | A metric evaluates the output of a model without assessing the process by which the output was generated, so a model that produces correct answers via incorrect or unsafe reasoning cannot be distinguished from one that reasons correctly | LLM achieves high accuracy on multiple-choice clinical questions by answer-pattern exploitation rather than clinical reasoning; pass@1 accuracy does not detect this | Chain-of-thought consistency test; adversarial reasoning probe |
| F6 | Performative Drift | Deployment | Latent | Deployment of the model changes the data-generating process in ways that invalidate the evaluation, because the evaluation was conducted on data from a world in which the model was not deployed | Sepsis prediction model evaluated on pre-deployment data; after deployment, clinicians modify treatment patterns in response to model alerts, shifting the outcome distribution | Temporal holdout; pre/post deployment distribution shift test |
| F7 | Fairness Impossibility | Normative | Latent | An evaluation specifies multiple fairness criteria that are mathematically mutually exclusive given the observed prevalence difference between groups, making simultaneous satisfaction impossible and the evaluation self-contradicting | Requiring equal false positive rates AND equal positive predictive value across groups with different base rates (Chouldechova's impossibility theorem) | Fairness criterion compatibility checker; impossibility boundary visualization |
| F8 | Sociotechnical Emergence | Normative | Latent | The evaluation measures the model in isolation from the human-AI system in which it will operate, missing failure modes that only emerge from the interaction between model outputs and clinical workflow | Diagnostic AI evaluated on images presented in isolation; in deployment, clinicians anchor on model predictions in ways that reduce their independent judgment, degrading the human-AI team's performance below the model's standalone accuracy | Human-in-the-loop simulation; automation bias probe |

### 1.2.1 Observations on the Taxonomy

Several structural patterns emerge from Table 1 that inform the design of
stratum-eval.

**Latency is the dominant detectability class.** Six of eight failure modes
are latent: they produce evaluation numbers that appear valid and that no
standard review process will flag. This means that neither higher sample
sizes nor stricter statistical testing will surface them. They require
targeted diagnostic tests that are not currently part of the standard
medical AI evaluation toolkit.

**Normative failures are not edge cases.** F7 and F8 are not exotic
pathologies. They arise in every multi-group evaluation that specifies more
than one fairness criterion (F7) and in every evaluation that treats model
performance as a proxy for system performance (F8). Their latency is
compounded by the fact that they are often invisible to the researchers
who designed the evaluation, because the normative commitments are implicit.

**The source axis predicts the type of intervention required.** Data failures
(F1, F3) require changes to how evaluation sets are constructed and
documented. Metric failures (F2, F4) require changes to which metrics are
selected and how they are interpreted. Deployment failures (F5, F6) require
changes to the evaluation protocol — specifically, closing the gap between
evaluation conditions and deployment conditions. Normative failures (F7, F8)
require changes to the value specification process that precedes evaluation
design.

stratum-eval addresses all four source categories. Its metric standards
(Section 2) primarily address F2 and F4. Its dataset auditing tools
(Section 3) primarily address F1 and F3. Its protocol recommendations
(Section 4) address F5 and F6. Its fairness compatibility checker
(Section 5) addresses F7 and F8.

## 1.3 Scope and Non-Goals

stratum-eval is an evaluation framework, not a model development framework.
It does not train models, tune hyperparameters, or recommend architectures.
It does not make deployment decisions; it surfaces information that
decision-makers need to make those decisions responsibly.

stratum-eval is also not a regulatory framework. It does not implement
FDA guidance, CE marking requirements, or any specific national standard
for AI as a medical device. Where our metric standards are consistent with
regulatory guidance (e.g., FDA's 2021 AI/ML action plan), we note this.
Where they diverge, our standards are stricter.

Finally, stratum-eval does not adjudicate the normative questions embedded
in F7 and F8. It surfaces impossibility constraints and workflow dependencies
so that they can be addressed explicitly rather than resolved implicitly by
metric choice.
